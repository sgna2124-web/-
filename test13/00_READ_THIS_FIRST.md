# TEST13 인수인계 시작점

이 폴더는 다음 대화의 어시스턴트가 바로 이어서 작업할 수 있도록 만든 상세 인수인계 문서 묶음이다.

가장 먼저 읽을 파일 순서:

1. `test13/01_project_state_and_objectives.md`
2. `test13/02_error_log_and_root_causes.md`
3. `test13/03_repo_file_map_and_attempt_history.md`
4. `test13/04_next_step_playbook.md`

핵심 결론만 먼저 적는다.

- 현재 목표는 여전히 동일하다. 차트 기반 자동매매 전략을 계속 개발한다.
- 성과 평가는 `MDD 5% 미만` 범위에서 `cd_value` 우선이다.
- 롱 전용, 숏 전용 전략을 별도로 계속 개발한다.
- 단순 파라미터 조정은 금지다. 구조적 개선, 필터 논리, 상태 판단, 진입/해제 로직 개선만 허용한다.
- 결과 보고 방식은 앞으로도 `전체 롱 전략 1위`와 `전체 숏 전략 1위`만 비교한다.
- 최근 실패는 전략 품질 문제가 아니라 실행 구조 문제였다. 특히 wrapper 설계, workflow YAML, 조합 파일 형식, 경로 일치 문제를 반복해서 터뜨렸다.
- 다음 어시스턴트는 먼저 실행 구조를 정상화한 뒤 전략 개발을 재개해야 한다. 전략 아이디어를 추가하기 전에 실행 계층을 안정화하는 것이 우선이다.

절대 반복하지 말아야 할 실수:

1. 상위 스크립트에 `run_batch`가 없는 계열을 wrapper에서 멋대로 `run_batch()`로 호출하지 말 것.
2. GitHub Actions workflow를 만들 때 heredoc, 들여쓰기, 불필요한 요약 스텝을 넣어 YAML 파싱 에러를 만들지 말 것.
3. workflow 이름은 사용자의 규칙을 따라 숫자부터 시작시킬 것.
4. 조합 파일 구조를 엔진 기대 형식과 다르게 쓰지 말 것.
5. 사용자가 이미 여러 번 확인한 사실, 특히 `세션의 핵심은 시간대 자체가 아니라 변동성 상태`라는 점을 무시하지 말 것.

이 문서 묶음은 실패 원인과 다음 액션까지 포함한다. 다음 작업자는 이 폴더를 기준으로 바로 복구 후 재개하면 된다.
