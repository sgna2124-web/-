# 프로젝트 상태와 목표

## 1. 최상위 목표

사용자의 장기 목표는 차트 기반 자동매매 전략을 계속 개선해서 실제로 돈을 벌 수 있는 구조를 찾는 것이다.
핵심은 OHLCV 기반 전략 개발이며, 최근 최대 약 1500캔들 범위에서 상태를 진단하고, 상태에 따라 전략을 선택하거나 비중을 조절하는 방향이 큰 흐름이다.

## 2. 현재 성과 평가 기준

사용자는 현재 성과 비교 기준을 아래처럼 고정했다.

- `MDD 5% 미만` 전략 우선
- 그 안에서 `cd_value`가 높은 순으로 우선순위 부여
- `cd_value`는 기본 자산에 MDD를 반영한 뒤 최대 수익률까지 반영하는 사용자 정의 지표
- 결과 보고는 앞으로 `전체 롱 전략 1위`, `전체 숏 전략 1위`만 알려주면 된다
- 새로운 백테스트 결과는 기존 전체 1위와만 비교한다

## 3. 작업 방식 제약

반드시 지켜야 할 제약은 아래와 같다.

1. 롱 전용 전략과 숏 전용 전략을 분리해서 개발할 것
2. 단순 파라미터 조정은 하지 말 것
3. 구조 자체를 바꾸는 실험을 할 것
4. GitHub Actions에서 workflow를 바로 실행할 수 있도록 할 것
5. 다음 단계 제안 시 이미 실패한 접근을 반복하지 말 것

## 4. 사용자가 반복해서 강조한 전략적 사실

### 4-1. 세션의 핵심은 시계 시간 자체가 아니다

사용자는 이미 과거 테스트를 통해 `세션의 중요성은 시간이 아니라 변동성 상태에 있다`는 점을 확인했다고 여러 번 언급했다.
따라서 단순히 UTC 시간대를 나눠 hot/quiet를 규정하는 방식만 반복하면 안 된다.
다음 어시스턴트는 시간 세션을 그대로 쓰더라도, 그것을 최종 해석 축이 아닌 임시 대체 축으로만 간주해야 한다.

### 4-2. 성공/실패 이유를 기반으로 장점만 결합해야 한다

사용자는 과거 전략들을 실제로 훑어보고, 성공/실패 이유를 분해해서 장점만 조합하라고 요구했다.
즉, 새 버전 번호만 늘리는 방식이 아니라 왜 어떤 구조가 먹히고, 왜 어떤 구조는 진입을 과도하게 죽이는지 분석해야 한다.

### 4-3. 보고 형식도 단순화되었다

이제는 매번 수많은 결과를 다 나열하지 않고, `전체 숏 1위`, `전체 롱 1위` 기준으로만 갱신 여부를 판단하면 된다.

## 5. 최근 전략 맥락 요약

최근 흐름에서 의미 있었던 축은 대략 아래와 같다.

- 숏 쪽은 `lane governor`, `hot/quiet`, `prior signal`, `dense timestamp 회피`, `reentry/entry gap 통제` 계열이 중요했다
- 롱 쪽은 `preserve prior`, `no reentry`, `hybrid guards`, `entry mode`, `prior signal 유지` 계열이 중요했다

다만, 최근 실패는 전략 논리에서 나온 것이 아니라 실행 계층에서 많이 발생했다. 따라서 다음 작업자는 전략 발상보다 실행 가능성 검증부터 해야 한다.

## 6. 다음 작업자가 반드시 먼저 해야 할 일

1. 현재 저장소에서 어떤 workflow가 실제로 문법상 유효한지부터 확인
2. 어떤 runner script가 실제로 독립 실행 가능한지 확인
3. 어떤 조합 파일이 엔진 기대 형식과 맞는지 확인
4. 그 다음에야 전략 개발 재개

실행 계층이 정리되지 않은 상태에서 새 버전을 계속 추가하면 다시 같은 실패가 난다.
