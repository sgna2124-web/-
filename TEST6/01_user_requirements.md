# 사용자 요구사항 및 금지사항

## 최종 목표
- OHLCV만으로 실전 운용 가능한 차트 전략을 찾는다.
- 레버리지를 사용할 수 있는 전략이어야 한다.
- 따라서 수익률뿐 아니라 MDD, 최대 동시 포지션 수, 자산 분할, 수수료까지 반드시 반영한다.
- 사용자는 복리 기준 실전 계좌 결과만 의미 있다고 본다.

## 사용자가 명시적으로 요구한 사항
1. 백테스트는 무조건 실전 전략 기반으로 산출할 것.
2. 동시에 보유하는 포지션 개수를 반드시 셀 것.
3. 최대 동시 포지션 수를 기준으로 자산 분할을 재설계할 것.
4. 수수료는 0.04%를 반영할 것.
5. 1시간봉 원본 데이터가 없으면 5분봉 12개로 합성해서 쓸 것.
6. 파이썬 백테스트 파일에서 print, plt, 엑셀 저장 등 사용자가 꾸며둔 구조는 손대지 말 것.
7. 수정은 진입 조건과 손익절 로직 중심으로 제한할 것.
8. 재현 가능한 숫자만 말할 것. 추정치, 감 숫자 금지.
9. 단일 종목과 전체 종목에서 일관되게 재현되는지 확인할 것.
10. 사용자는 MDD 30% 수준은 감당 가능하다고 보지만, 그 이상은 레버리지 운용에 부담이 크다고 본다.

## 동시 포지션 수 기반 자산 분할 원칙
- 임의의 1% 고정 리스크를 최종 기준으로 삼지 말 것.
- 먼저 전체 백테스트에서 최대 동시 보유 포지션 수를 측정할 것.
- 그 최대 개수만큼 자산을 분할하는 방식으로 재백테스트할 것.
예시:
- 최대 동시 포지션 73개 -> 약 1/73 = 1.37% 수준 검토
- 최대 동시 포지션 200개 -> 0.5% 수준 검토
핵심:
- 최대 동시 포지션 수 계산 -> 포지션당 자산 비중 재설정 -> 전체 재백테스트 순서를 반드시 지킬 것

## 사용자가 싫어하는 것
- OHLCV만으로는 구조 데이터를 못 보니 못 번다 같은 뻔한 말
- 단일 지표형 전략
- 확인되지 않은 고수익 숫자를 사실처럼 말하는 것
- 구현 오류를 잡기 전에 수치 완화부터 하는 것
- 실전과 무관한 백테스트 방식

## 절대 반복하면 안 되는 행동
- 재현 안 되는 고수익 결과를 계속 기준점처럼 사용하는 것
- 전체 백테스트도 안 끝났는데 최적 전략이라고 단정하는 것
- 1시간 필터를 룩어헤드 가능성이 있는 방식으로 구현하는 것
- 단일 종목 검증 없이 전체 종목 결과를 확정하는 것
