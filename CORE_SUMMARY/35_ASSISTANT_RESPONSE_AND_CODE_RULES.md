어시스턴트 응답 및 코드 산출 행동 수칙

이 문서는 새 대화창의 어시스턴트가 반드시 적용해야 하는 응답 방식과 파이썬 코드 산출 규칙이다.
CORE_SUMMARY/09_NEXT_CHAT_MANUAL.md 와 함께 적용한다.

1. 응답 방식
대답은 감정적 공감보다 핵심 전달을 우선한다.
불필요한 위로, 변명, 장황한 배경 설명을 줄인다.
사용자가 저장소 확인, 결과 분석, 코드 작성, 문서 갱신을 지시하면 다음 네 가지만 우선 보고한다.
1) 확인한 대상
2) 판단한 핵심 결론
3) 실제 수정 또는 생성한 파일
4) 다음 실행에 필요한 최소 지침

2. 파이썬 코드 산출 기본 원칙
파이썬 코드를 작성한 경우, 사용자가 특별히 다른 방식을 요청하지 않는 한 현재 대화창에 .py 파일로 만들어 업로드한다.
긴 코드는 채팅 본문에 전부 붙여넣지 말고, 파일로 제공한 뒤 본문에는 실행 명령과 핵심 변경점만 적는다.
저장소에 바로 커밋하는 것은 사용자가 저장소 반영을 명시했을 때만 수행한다.

3. 업로드 전 코드 점검 원칙
파이썬 파일을 업로드하기 전에 전체적으로 오류 가능성을 점검한다.
최소 점검 항목은 다음과 같다.
1) py_compile 수준 문법 오류
2) import 누락
3) 함수명, 변수명, 클래스명 오타
4) 정의되지 않은 FeaturePack 필드 참조
5) 정의되지 않은 StrategyParams 필드 참조
6) 정의되지 않은 entry_signal 또는 mask 참조
7) pandas/numpy 배열의 truth value ambiguity 가능성
8) UnboundLocalError 가능성이 있는 조건부 변수 할당
9) 사용자 로컬 경로, 결과 폴더명, 저장 파일명 불일치
10) 기존 백테스트 파일의 고정 환경 훼손 여부
11) TP 기대값 0.3% 이상 진입 필터 적용 여부
12) 사용자가 제공한 로컬 경로의 의미를 올바르게 분류했는지 여부
13) 기준선 통과 파일을 재구성하지 않았는지 여부
14) 결과 저장 경로 강제값이 실제 저장 함수에서 사용되는지 여부
15) 결과 저장 경로가 로그에 출력되고 실행 초기에 assert로 검증되는지 여부

4. 실제 실행 검증 기준
가능하면 업로드 전 py_compile을 통과시킨다.
외부 데이터, 사용자 로컬 경로, 대용량 OHLCV 파일이 없어 전체 백테스트 실행이 불가능한 경우에도 문법 검사와 정적 참조 점검은 수행한다.
실행 검증을 못 한 부분이 있으면 무엇을 못 했는지만 짧게 명시하고, 검증 가능한 오류는 사전에 제거한다.

5. 이 프로젝트에서 특히 금지할 반복 오류
정의되지 않은 로컬 변수 사용 금지.
numpy/pandas 배열을 if 조건에 그대로 넣는 구조 금지.
FeaturePack에 없는 속성을 전략 조건에서 호출하는 구조 금지.
StrategyParams에 없는 필드를 전략 로직에서 참조하는 구조 금지.
기존 공식 백테스트 환경인 자산 진입 비중 1%, 수수료 0.04%, 전체 종목 풀 기준을 코드 생성 과정에서 임의로 바꾸는 행위 금지.
파이썬 백테스트 환경은 깃허브 저장소 환경과 다르다.
사용자의 실제 로컬 GitHub 저장소 접근 경로는 C:\Users\user\Documents\GitHub\- 이다.
이 경로는 저장소 접근 경로이지 파이썬 실행 위치가 아니다.
이 경로를 cwd, 실행 위치, 작업 디렉터리로 단정하지 않는다.
사용자가 별도로 실행 위치를 말하지 않는 한 실행 위치는 알 수 없는 값으로 둔다.
TP 기대값이 0.3% 이상일 때만 진입하는 조건을 모든 신규 백테스트 개선안에 반드시 포함한다.
기준선이 통과된 파일을 근거 없이 재구성하지 않는다.
기준선 조건, base_spec, BASELINES, entry/exit 엔진, parent gate는 사용자가 명시적으로 허용하지 않는 한 수정하지 않는다.

6. 전략 개선안 필수 진입 조건
앞으로 생성하는 모든 백테스트 개선안은 TP 기대값이 0.3% 이상일 때만 진입해야 한다.
코드상으로는 최소 기대 TP 필터를 min_expected_tp >= 0.003 또는 동등한 조건으로 명시한다.
롱/숏 기준선 자체를 검증하는 복제 전략은 원본 기준선 보존을 우선하되, 기준선 통과 후 생성되는 개선안에는 반드시 이 조건을 적용한다.
해당 조건을 적용하지 않은 개선안은 전략 후보로 생성하지 않는다.

7. 기준 파일 보존 및 실패 후 행동 규칙
사용자가 “기준 파일을 최대한 훼손하지 말라”, “기준선 전략 조건은 손대면 안 된다”, “CORE_SUMMARY 안의 파이썬 파일이 기준 파일이다”라고 말하면 다음을 반드시 지킨다.
1) 기준 파일을 새로 재작성하지 않는다.
2) 원본 기준 파일의 BASELINES, base_spec, 기준선 4개 조건, entry/exit 엔진, parent gate를 수정하지 않는다.
3) 개선안 생성이 필요하면 원본 파일을 복사한 뒤 variants_for_axis 등 개선안 생성부만 최소 수정한다.
4) 기준선 gate가 실패하면 즉시 해당 파일을 폐기 대상으로 보고, 같은 방식으로 재시도하지 않는다.
5) 실패 원인이 로컬 경로 오해, 기준선 재구성, 원본 엔진 훼손 중 무엇인지 분류한다.
6) 같은 실수를 반복하지 않도록 CORE_SUMMARY에 재발 방지 규칙을 먼저 등록한 뒤 다음 코드를 만든다.
7) 사용자가 제공한 C:\Users\user\Documents\GitHub\- 는 저장소 접근 경로로만 사용하고, 실행 위치로 취급하지 않는다.

8. 결과 저장 경로 강제 규칙
사용자가 결과 저장 경로를 C:\Users\user\Documents\GitHub\-\local_results 로 지정하면 백테스트 결과는 반드시 이 폴더 하위에 저장한다.
BATCH별 결과 폴더는 C:\Users\user\Documents\GitHub\-\local_results\{BATCH_NAME} 구조를 사용한다.
코드에 LOCAL_RESULTS_ROOT 상수를 선언하는 것만으로는 충분하지 않다. 실제 저장 변수(out_dir, outdir, result_dir, results_dir, save_dir 등)가 이 강제 경로를 사용하도록 main 실행부에서 직접 할당해야 한다.
argparse에 결과 경로 옵션이 없거나 parse_args 강제 패치가 실패할 수 있으므로, 저장 직전의 최종 out_dir 변수 자체를 LOCAL_RESULTS_ROOT / BATCH 로 지정한다.
실행 시작 로그에는 반드시 [RESULTS_DIR] 또는 동등한 출력으로 실제 저장 경로를 표시한다.
실행 초기에 str(out_dir).lower()가 str(LOCAL_RESULTS_ROOT).lower()로 시작하는지 assert하거나, 불일치하면 RuntimeError를 발생시킨다.
경로 보정 패치 후에는 문자열 치환 여부가 아니라 실제 저장문 CSV/JSON/TXT가 참조하는 최종 디렉터리를 확인한다.
