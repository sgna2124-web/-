로컬 파이썬 백테스트 환경 규칙

가장 중요한 확정 사항
사용자의 실제 로컬 GitHub 저장소 접근 경로는 아래 폴더 계층이다.

폴더 계층:
1. C:
2. Users
3. user
4. Documents
5. GitHub
6. -

문자 단위 경로:
C: + 역슬래시 + Users + 역슬래시 + user + 역슬래시 + Documents + 역슬래시 + GitHub + 역슬래시 + 하이픈

정확한 의미:
Documents 아래에 GitHub 폴더가 있고, 그 GitHub 폴더 안에 이름이 - 인 저장소 폴더가 있다.
Documents 아래에 GitHub- 라는 폴더가 있는 것이 아니다.

틀린 구조:
Documents 아래에 GitHub- 폴더가 있는 구조

맞는 구조:
Documents 아래에 GitHub 폴더가 있고, GitHub 폴더 안에 - 폴더가 있는 구조

중요한 해석
이 경로는 사용자의 컴퓨터에서 깃허브 저장소 파일에 접근하는 경로다.
이 경로는 파이썬 파일을 실행하는 위치가 아니다.
앞으로 이 경로를 설명할 때 절대 실행 위치, 작업 디렉터리, cwd, 파이썬 실행 폴더로 단정하지 않는다.
사용자가 별도로 실행 위치를 말하지 않는 한, 실행 위치는 알 수 없는 값으로 둔다.

경로 표기 고정 규칙
1. 경로를 설명할 때는 “GitHub 폴더 안의 - 폴더”라고 함께 설명한다.
2. GitHub와 - 사이에는 폴더 구분자가 있다.
3. GitHub와 -를 붙여 쓰면 오답이다.
4. GitHub 폴더 안에 - 폴더가 있다는 계층 구조를 우선으로 기억한다.
5. 슬래시 방향을 임의로 바꾸지 않는다.
6. 이 경로는 저장소 접근 경로이며, 실행 위치가 아니다.
7. 답변에서 경로가 Markdown에 의해 흐려질 가능성이 있으면 문자 단위 경로 또는 폴더 계층으로 설명한다.
8. 일반 문장 안에서 마지막 “역슬래시 + 하이픈” 조합이 화면에서 하이픈만 남아 보일 수 있으므로, 단순 문자열만으로 경로를 설명하지 않는다.

파이썬 백테스트 환경은 깃허브 저장소 도구 환경과 다르다.
어시스턴트가 GitHub 도구로 보는 저장소 환경과 사용자가 Windows PowerShell에서 실행하는 로컬 파이썬 환경은 같다고 가정하면 안 된다.

적용 원칙
1. 로컬에서 실행될 백테스트 .py 파일은 실행 중 GitHub 웹/도구 전용 경로를 import, open, read_json, glob 방식으로 참조하면 안 된다.
2. 사용자의 로컬 저장소 접근 경로는 GitHub 폴더 안의 - 폴더로만 이해한다.
3. 사용자의 로컬 저장소 접근 경로를 파이썬 실행 위치로 가정하지 않는다.
4. 사용자가 실제 실행 위치를 알려주지 않았다면, 실행 명령은 특정 cwd를 강제하지 않는 방식으로 안내한다.
5. 기준선 전략 조건, 참조값, 조합 설정이 필요하면 코드 생성 시점에 확인해 단일 실행 파일 내부에 직접 탑재한다.
6. 단, 기준선이 이미 통과된 기준 파일이 있을 때는 기준선 조건과 엔진을 재구성하지 않는다. 원본 기준 파일을 최대한 보존하고, 개선안 생성부만 최소 수정한다.
7. 결과 대조용 코드도 GitHub 도구 환경의 경로를 로컬에서 찾게 만들지 않는다. 필요한 참조값은 파일 내부에 내장하거나, 사용자의 로컬 저장소 접근 경로 아래의 명시 파일로 안내한다.
8. 저장소 파일을 수정하거나 확인하는 작업은 어시스턴트가 GitHub 도구로 수행하고, 사용자의 로컬 파이썬 실행 과정에는 의존시키지 않는다.
9. 로컬 파일을 생성하라고 할 때는 GitHub 폴더 안의 - 폴더 아래 어느 파일을 기준으로 삼는지 명확히 말한다.
10. 코드 파일을 업로드할 때는 사용자가 저장소 접근 경로 아래에 복사할 수 있는 독립 파일로 제공하거나, 원본 파일을 로컬에서 읽어 최소 패치하는 방식으로 제공한다.

반복 실수 방지 규칙
1. 사용자가 기준 파일을 훼손하지 말라고 말하면, 기준선 조건, base_spec, BASELINES, entry/exit 엔진, parent gate를 재작성하지 않는다.
2. 기준선이 통과된 파일이 이미 CORE_SUMMARY 안에 있으면, 그 파일을 새로 재구성하지 않는다.
3. 기준 파일을 바탕으로 개선안을 만들 때는 원본 파일 전체를 복사한 뒤 variants_for_axis 같은 개선안 생성부만 최소 수정한다.
4. 기준선 gate가 실패하면 같은 방식으로 다시 코드를 만들지 않는다. 먼저 실패 원인을 원본 보존 실패 또는 로컬 경로/실행 환경 오해로 분류하고 CORE_SUMMARY에 재발 방지 규칙을 업데이트한다.
5. 사용자가 로컬 경로를 알려주면, 그 경로의 의미를 먼저 구분한다. 접근 경로인지, 실행 위치인지, 데이터 위치인지, 결과 위치인지 혼동하지 않는다.
6. 저장소 접근 경로는 현재까지 “GitHub 폴더 안의 - 폴더”로만 확정되어 있다.
7. 경로를 복사하거나 설명할 때는 GitHub와 - 사이에 폴더 구분자가 있음을 반드시 확인한다.
8. GitHub와 -를 붙여 쓰는 경로는 오답이다.
