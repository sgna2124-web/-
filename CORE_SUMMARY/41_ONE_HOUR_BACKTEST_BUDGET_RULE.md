1시간 예산형 백테스트 개선 규칙

작성 배경
8V24 실행에서 processed=25/597, elapsed=7139.4s가 기록되었다. 개선안 개수를 4축 50개씩 맞추는 방식은 실전 반복 개선에 부적합하다. 앞으로는 개선안 개수보다 1시간 내외에 검증 가능한 실행량을 우선한다.

핵심 행동 수칙
1. 앞으로 개선 파일을 만들 때 개선안 개수는 중요하지 않다.
2. 목표는 1시간 내외에 백테스트 가능한 데이터 양이다.
3. 사용자가 몇 개라고 명시하지 않은 경우, 1시간 내외 완료 가능성을 우선해 축 수, max-bars, 개선안 수를 역산한다.
4. 1시간을 크게 넘길 가능성이 있으면 개선안 수를 줄이는 것이 아니라 데이터량, 축 범위, max-bars, 실험 축을 줄인다.
5. processed/elapsed 로그가 있으면 반드시 이를 기준으로 다음 배치 크기를 계산한다.
6. 빠른 반복 검증이 목적이므로 처음부터 전체 597종목, max-bars 1500, 대량 개선안을 고정하지 않는다.
7. 1시간 예산형에서 살아남은 후보만 max-bars와 데이터 범위를 단계적으로 늘린다.
8. 1시간 예산형 결과만으로 공식 승격하지 않는다. 공식 승격은 확장 검증 후 판단한다.

새 기준
1. 백테스트 개선 배치는 개수 기준이 아니라 시간 예산 기준으로 설계한다.
2. 기본 목표는 1시간 내외에 완료 가능한 실행량이다.
3. 전체 종목 597개와 max-bars 1500을 고정한 채 200개 개선안을 한 번에 돌리는 방식은 금지한다.
4. 1시간 예산형 기본값은 각 축 1개 이하 또는 선택 축 1개 단위로 시작한다.
5. max-bars는 필요 시 600~800 수준부터 시작하고, 결과가 유의미하면 1000, 1200, 1500 순으로 확장한다.
6. 시간 초과가 발생하면 개선안 수를 늘리는 것이 아니라 데이터량, 축 범위, max-bars를 줄인다.
7. 실행 로그의 processed, elapsed를 보고 다음 배치 크기를 역산한다.
8. 기준선 조건, base_spec, BASELINES, entry/exit 엔진, parent gate는 계속 보존한다.
9. TP 기대값 0.3% 이상 진입 조건은 신규 개선안에 계속 적용한다.
10. 결과 저장 경로는 C:\Users\user\Documents\GitHub\-\local_results\{BATCH} 를 유지한다.
11. subprocess cwd는 사용자의 launch_cwd를 유지하고, 저장소 접근 경로를 실행 위치로 강제하지 않는다.

시간 예산 역산 규칙
- elapsed / processed 로 종목 1개당 평균 처리 시간을 구한다.
- 예상 총 시간 = 종목 수 * 축 수 * 후보 수 보정치 * 평균 처리 시간으로 대략 추정한다.
- 1시간을 넘길 가능성이 높으면 즉시 후보 수, 축 수, max-bars 중 하나를 줄인다.
- 사용자가 시간 제한을 언급하면 개수 목표보다 시간 목표를 우선한다.
- 과거처럼 50개, 100개, 400개 같은 개수 목표를 자동으로 유지하지 않는다.
