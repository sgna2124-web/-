# 반드시 지켜야 할 원칙 / 엔진 검증 메모

## 1. 절대 원칙

1. 모든 백테스트는 실제 시간축 기반이어야 한다.
   - 이 원칙은 사용자가 매우 강하게 요구한 사항이다.
   - 같은 시각의 entry/exit 처리 순서, 포트폴리오 time-axis 평가 방식, realized equity 추적이 중요하다.

2. 수수료는 fee_per_side = 0.0004를 유지한다.
   - 왕복 기준 0.08%.
   - 어떤 새 실험이든 이 값이 빠지면 비교 무효다.

3. pre-v22 상위 성과를 공식 기준으로 재사용하면 안 된다.
   - v22 이전에는 same-bar 처리 누수 문제 때문에 max_conc와 수익이 비정상적으로 커졌던 구간이 있다.
   - 사용자는 이 문제에 매우 민감하다. 다시 같은 실수를 반복하면 안 된다.

4. max_conc가 비정상적으로 커지면 즉시 성과 개선 실험을 멈추고 엔진/집계부터 다시 검증한다.
   - “성과가 좋으니 일단 넘어간다” 금지.
   - 이상치가 보이면 먼저 오류 가능성을 의심한다.

5. 현재 top200은 공식 기준선일 뿐, 최종 배포 규칙으로 확정된 것이 아니다.
   - 성과 기준으로는 현재까지 최고.
   - 그러나 구조적 근거가 약하고, 실거래 구현 관점에서도 사용자가 거부감을 표했다.

## 2. 실제로 발생했던 치명적 오류

### 오류 A: 실제 시간축이 아닌 잘못된 평가/해석 구간
사용자가 중간에 직접 지적했고, 이후부터는 반드시 실제 time-axis 기준 평가만 사용하도록 수정했다.
이전의 일부 결과는 비교 참고용으로만 보고, 공식 기준은 revalidated 이후만 사용한다.

### 오류 B: same-timestamp entry/exit 처리 누수
이게 더 치명적이었다.
이전 엔진에서는 같은 타임스탬프에서 entry_ts와 exit_ts가 같은 거래를 처리할 때 active 포지션이 정상적으로 제거되지 않아 max_conc와 성과가 과장되었을 가능성이 컸다.

결과적으로 생긴 현상:
- max_conc 33539 같은 비정상 수치
- 수익률이 과도하게 좋아 보이는 결과
- 종목 수 상한과 논리적으로 맞지 않는 동시 보유수

v22 revalidation에서 same-bar 누수 수정 후:
- long_guard_wick_only_top200_baseline은 513.259% / MDD -48.724 / max_conc 234로 크게 내려옴
- atr18975_t27_b17_r155_legacy_baseline은 1082.019% / MDD -40.719 / max_conc 386으로 raw 1위가 됨

즉, pre-v22 / pre-revalidation 상위 순위는 상당 부분 재해석이 필요했다.

## 3. 현재 공식적으로 믿을 수 있는 엔진 범위

공식적으로 신뢰하는 범위:
- v22 strict revalidate 이후
- same-bar 처리 수정 엔진
- 실제 time-axis 기반 포트폴리오 평가
- fee_per_side=0.0004 유지

공식 비교선으로 허용하는 버전 흐름:
- v22: revalidation 출발점
- v23: baseline_top200 승격
- v24: score/topN/MDD/CD 실험
- v25: top200 진단
- v26: hardened absolute score 1차 시도
- v27: top200 제거 대체 시도

## 4. max_conc 해석 주의

- max_conc는 현재 엔진에서 “포트폴리오 전체 활성 포지션 수 최대치”다.
- 같은 심볼 내 중첩 진입이 아니라, 여러 심볼이 동시에 살아 있는 개수다.
- 같은 심볼에서는 inpos 플래그 때문에 기존 포지션이 종료되기 전 재진입하지 않는다.
- 다만 자본 총량/증거금 총량 제한은 아직 엄격히 모델링되지 않았으므로, max_conc 해석은 여전히 신중해야 한다.

## 5. 다음 어시스턴트가 절대 잊지 말아야 할 것

- “성과가 좋아 보인다”는 이유로 검증을 건너뛰지 말 것
- 이상한 max_conc, 비정상 수익률, 설명 안 되는 MDD가 나오면 우선 엔진/집계를 검토할 것
- top200은 현재 최선의 기준선이지만, 구조적으로 확정된 정답으로 간주하지 말 것
- 사용자는 실거래 구현 가능성과 논리성을 매우 중요하게 본다
- 점점 더 “설명 가능한 규칙”을 원한다
